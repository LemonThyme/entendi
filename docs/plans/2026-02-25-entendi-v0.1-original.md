PRODUCT SPECIFICATION

**Entendi**

The Comprehension Accountability Layer

*An MCP-based middleware that ensures developers, students, and
researchers*

*actually understand what their AI tools are doing for them.*

v0.1 Draft --- February 2026

CONFIDENTIAL

1\. Executive Summary

**Entendi** is an MCP server that acts as a **comprehension accountability
layer** for AI-assisted work. It monitors how individuals interact with
LLM tools, periodically probes their understanding through Socratic
questioning, and produces a persistent **knowledge graph** that maps
what someone actually understands versus what they have merely
copy-pasted or delegated.

The core philosophy is permissive, not restrictive: use any AI tool you
want. The only requirement is that you can demonstrate you understand
what those tools did for you and why you asked for it.

+-----------------------------------------------------------------------+
| **The Problem**                                                       |
|                                                                       |
| Anthropic's own research (2025) found that developers using AI        |
| assistance scored 17% lower on comprehension quizzes --- equivalent   |
| to nearly two letter grades. The failure mode isn't using AI; it's    |
| using AI as a black-box pass-through without building any mental      |
| model.                                                                |
+-----------------------------------------------------------------------+

**Target users:** Universities, coding bootcamps, engineering
organizations, and research labs that want to allow AI tool usage while
ensuring comprehension and skill development.

2\. Problem Space

2.1 The Comprehension Crisis

Current approaches to managing AI tool usage in educational and
professional settings fall into three inadequate categories:

-   **Ban it:** Prohibiting AI tools entirely. Unenforceable and
    counterproductive --- these tools are the future of work.

-   **Ignore it:** Allowing unrestricted use with no accountability.
    Leads to skill atrophy, especially among junior developers and
    students.

-   **Detect it:** Using AI-detection tools after the fact (Turnitin,
    GPTZero). High false-positive rates, adversarial arms race, and
    doesn't address the actual problem of comprehension.

None of these approaches answer the question that actually matters: does
this person understand the work they produced?

3\. Product Overview

3.1 What Entendi Is

Entendi is an MCP server (or set of MCP servers) that organizations install
alongside their developers'/students' AI tooling. It provides three core
capabilities:

1.  **Observation:** Passively monitors all LLM interactions --- prompts
    sent, code generated, repos installed, files created, tools invoked.

2.  **Interrogation:** Periodically and contextually probes the user's
    understanding through Socratic questioning, calibrated to the
    novelty and complexity of what they're working on.

3.  **Reporting:** Produces a persistent knowledge graph and
    comprehension scores visible to the individual, their
    manager/professor, and the organization.

3.2 What Entendi Is Not

-   Not a replacement for existing AI tools (Copilot, Claude, Cursor,
    etc.)

-   Not a plagiarism detector or AI-content classifier

-   Not a restrictive gatekeeper that blocks tool usage

-   Not a surveillance tool --- it measures comprehension, not
    productivity or time-on-task

3.3 Core Design Principles

4.2 Entendi Observer

The Observer is the core MCP server that must be installed on every
user's development environment. It operates as a transparent proxy or
sidecar:

4.2.1 What It Captures

-   **Prompts:** Every query sent to an LLM, with context about what
    file/project is active.

-   **Responses:** The LLM's output --- code generated, explanations
    given, errors diagnosed.

-   **Tool Invocations:** Package installations (npm install, pip
    install), repo clones, MCP tool calls.

-   **File Mutations:** What files were created, modified, or deleted as
    a result of AI-assisted work.

-   **Acceptance Patterns:** Did the user accept the suggestion
    verbatim? Modify it? Reject and rewrite?

4.2.2 Novelty Detection

The Observer classifies each interaction by novelty relative to the
user's established knowledge graph:

  ---------------------------------------------------------------------------
  **Level**               **Description**             **Probe Frequency**
  ----------------------- --------------------------- -----------------------
  Routine                 User has demonstrated       \~5% of interactions
                          understanding of this       
                          concept/tool before         

  Adjacent                Related to known concepts   \~25% of interactions
                          but extends into new        
                          territory                   

  Novel                   Entirely new concept,       \~60% of interactions
                          library, pattern, or domain 

  Critical                Security-sensitive,         \~80% of interactions
                          infrastructure-affecting,   
                          or architecturally          
                          significant                 
  ---------------------------------------------------------------------------

4.3 Entendi Interrogator

  ----------------------- ----------------------- -----------------------

4.3.3 Anti-Gaming Measures

Since users could ask another LLM for answers, Entendi employs several
countermeasures:

-   **Latency analysis:** If a user consistently takes 30+ seconds to
    answer simple probes (suggesting they're querying another LLM), this
    is flagged.

-   **Follow-up chains:** Probes aren't one-shot. A good answer triggers
    a deeper follow-up. Parroted answers from LLMs tend to break down at
    depth 2--3.

-   **Context-specific probes:** Questions reference specific details of
    the user's codebase and current state that would require significant
    context to feed to another LLM.

-   **Consistency checks:** The knowledge graph tracks understanding
    over time. A user who claims deep Redis knowledge in a probe but
    then asks basic Redis questions to their LLM creates a detectable
    inconsistency.

-   **Oral defense triggers:** For high-stakes work (production
    deployments, thesis submissions), Entendi can flag for a human oral
    review based on knowledge graph gaps.

4.4 Entendi Knowledge Engine

The Knowledge Engine maintains a persistent, evolving graph of each
user's demonstrated understanding.

4.4.1 Graph Structure

The knowledge graph is a directed graph where:

-   **Nodes** represent concepts, tools, libraries, patterns, and
    domains (e.g., "async/await", "Redis caching", "B-tree indexing",
    "Docker networking").

-   **Edges** represent relationships: prerequisites, alternatives,
    compositions (e.g., "Redis caching" requires "key-value stores",
    is-alternative-to "Memcached").

-   **Node weights** represent the user's demonstrated comprehension
    level (0--3) for each concept, with a time-decay function so stale
    knowledge degrades.

-   **Interaction logs** are attached to nodes: every time a concept is
    touched (via LLM interaction or probe response), it's recorded.

4.4.2 Concept Extraction

Concepts are extracted from LLM interactions using a combination of:

1.  **Static analysis:** Parse imports, package.json, requirements.txt,
    Dockerfiles, etc. to identify tools and libraries in use.

2.  **Semantic analysis:** Use an LLM to extract conceptual topics from
    prompts and responses (e.g., a prompt about "optimistic locking"
    touches concurrency, database transactions, and conflict
    resolution).

3.  **Taxonomy mapping:** Map extracted concepts to a shared
    organizational taxonomy so that different users' knowledge graphs
    are comparable.

4.5 Entendi Dashboard

4.5.1 Individual View

Each user sees their own knowledge graph as an interactive
visualization:

-   Heat-mapped nodes showing strong (green), developing (yellow), and
    weak (red) understanding areas

-   Timeline of comprehension growth per concept

-   Recommended learning paths based on identified gaps

-   History of probe interactions and their evaluations

4.5.2 Manager/Professor View

-   Aggregated team/class knowledge map showing collective strengths and
    gaps

-   Individual comprehension scores with drill-down capability

-   Flags for high-risk patterns: heavy delegation without
    understanding, knowledge regression, inconsistencies

-   Comparison views: knowledge demonstrated vs. code shipped (the
    "comprehension debt" ratio)

4.5.3 Organization View

-   Cross-team knowledge distribution analysis

-   Institutional knowledge risk: concepts understood by only one person

-   Onboarding effectiveness: how quickly new hires build their
    knowledge graphs

-   AI tool ROI: are AI tools accelerating learning or replacing it?

5\. User Experience Flows

5.1 Developer Daily Flow

A typical interaction cycle from the developer's perspective:

1.  **Work normally:** Developer uses Cursor/Copilot/Claude Code as
    usual. Entendi Observer runs silently in the background.

2.  **Contextual probe appears:** After installing a new npm package, a
    small non-blocking notification appears: "Entendi: You just added zod.
    Quick question --- why schema validation at the boundary rather than
    deeper in the stack?"

3.  **Developer responds:** Types a brief answer in the Entendi
    sidebar/chat panel. Takes 30--60 seconds.

4.  **Entendi evaluates and adapts:** Good answer? Entendi notes "zod" and
    "schema validation" as Level 2 understanding. Weak answer? Entendi
    notes the gap and will probe again when the concept recurs.

5.  **Continue working:** Next probe might not come for 30 minutes if
    the developer is in familiar territory.

+-----------------------------------------------------------------------+
| **Design Goal: Minimal Friction for the Competent**                   |
|                                                                       |
| A developer who genuinely understands their work should barely notice |
| Entendi. Probes should feel like a brief, natural checkpoint --- not an  |
| exam. The friction should be proportional to the gap between what     |
| they're doing and what they've demonstrated they know.                |
+-----------------------------------------------------------------------+

5.2 Student Course Flow

For educational contexts, the flow is similar but with tighter
integration into course structure:

-   Professor defines which concepts the course covers and maps them to
    the taxonomy

-   Students work on assignments using any AI tools they want, with Entendi
    installed

-   Entendi probes are calibrated to the course's expected learning
    outcomes

-   At assignment submission, the student's knowledge graph for relevant
    concepts is attached alongside their work

-   Grading considers both the output quality and the demonstrated
    comprehension

-   Students can see their own growth over the semester --- creating
    intrinsic motivation to actually learn

7\. Privacy, Ethics & Governance

Entendi captures sensitive data about how people think and work. This
section is not an afterthought --- it's a foundational design
constraint.

7.1 Data Ownership

-   **Individual data belongs to the individual.** Users can export or
    delete their knowledge graph at any time.

-   **Organizations see aggregated views by default.** Drill-down to
    individual probes requires explicit policy agreement.

-   **Interaction logs (raw prompts/responses) are never shared with
    managers.** Only derived comprehension scores and knowledge graph
    states are reportable.

-   **Right to explanation:** Users can see exactly why a concept was
    scored at a particular level and challenge it.

7.2 Anti-Surveillance Safeguards

-   Entendi measures comprehension, not effort, time, or productivity

-   No keystroke logging, screen recording, or mouse tracking

-   Probe responses are evaluated for understanding, not for speed or
    volume of work

-   Managers cannot see what prompts a user sent to their AI tools ---
    only what concepts were involved and the comprehension assessment

-   Organizations must define a written policy for how Entendi data is used
    before deployment

7.3 Ethical Considerations

-   **Neurodivergence:** Probe delivery and response formats must be
    configurable. Some users may prefer written responses, others voice.
    Time pressure on responses should be avoided.

-   **Language:** Non-native speakers should not be penalized for
    expression quality. Probes evaluate conceptual understanding, not
    linguistic fluency.

-   **Anxiety:** Framing matters. Entendi should never feel like a test.
    The UX should emphasize growth tracking, not evaluation.

-   **Power dynamics:** Clear policies must prevent Entendi data from being
    used in performance reviews without the employee's explicit consent
    and context.

10\. Open Questions

1.  **Evaluation model:** Should Entendi use the same LLM provider the user
    is working with (better context) or a separate one (independence)?
    There's a conflict-of-interest argument for separation.

2.  **Probe intrusiveness calibration:** How do we find the right
    balance between probing enough to be useful and probing so much that
    it impedes real work? This likely needs A/B testing across user
    populations.

3.  **Gaming surface area:** If the user has a second device, they can
    always feed probe questions to another LLM. How aggressive should
    anti-gaming be before it becomes adversarial? The oral-defense
    trigger for high-stakes work may be the most practical solution.

4.  **Organizational taxonomy design:** Who defines the concept
    taxonomy? Should it be crowd-sourced from the knowledge graphs
    themselves, defined top-down by leadership, or both?

5.  **Regulatory landscape:** In some jurisdictions, workplace
    monitoring (even comprehension monitoring) may require works council
    approval or individual consent. Legal review needed per-market.

6.  **Integration depth vs. vendor lock-in:** Deep IDE integration
    provides better observation but creates maintenance burden. MCP as
    the integration layer helps, but the protocol is still evolving.
