**Entendi**

The Comprehension Accountability Layer

*An adaptive middleware that ensures developers, students, and researchers
actually understand what their AI tools are doing for them.*

v0.2.1 Revised Draft --- February 2026

CONFIDENTIAL

1\. Executive Summary

**Entendi** is a comprehension accountability layer for AI-assisted work. It
observes how individuals interact with LLM tools, probes their understanding
through Socratic questioning, optionally teaches them through adaptive
tutoring, and maintains a **Bayesian knowledge graph** that estimates what
someone actually understands versus what they have merely copy-pasted or
delegated.

The knowledge graph is not a scorecard with integer ratings. It is a
probabilistic model: each concept carries a posterior distribution over the
user's mastery, updated via Bayesian inference from probe responses and
decayed over time using spaced repetition mathematics. The system can
estimate not just what you know, but how confident it is in that estimate,
what you're ready to learn next, and when it should re-assess you.

The core philosophy is **permissive, not restrictive**: use any AI tool you
want. The only requirement is that you can demonstrate you understand what
those tools did for you and why you asked for it.

**Target users:** Universities, coding bootcamps, engineering organizations,

ensuring comprehension and skill development.

2\. Problem Space

2.1 The Comprehension Crisis

Current approaches to managing AI tool usage in educational and professional
settings fall into three inadequate categories:

-   **Ban it:** Prohibiting AI tools entirely. Unenforceable and
    counterproductive --- these tools are the future of work.

-   **Ignore it:** Allowing unrestricted use with no accountability. Leads
    to skill atrophy, especially among junior developers and students.

-   **Detect it:** Using AI-detection tools after the fact (Turnitin,
    GPTZero). High false-positive rates, adversarial arms race, and doesn't
    address the actual problem of comprehension.

None of these approaches answer the question that actually matters: **does
this person understand the work they produced?**

3\. Product Overview

3.1 What Entendi Is

Entendi is a set of tool-specific adapters and services that organizations
install alongside their developers'/students' AI tooling. It provides four
core capabilities:

1.  **Observation:** Monitors LLM interactions --- prompts sent, code
    generated, tools invoked, files mutated --- through per-tool adapters
    (Claude Code hooks, IDE extensions, API proxies).

2.  **Interrogation:** Contextually probes the user's understanding through
    Socratic questioning, calibrated to the novelty and complexity of what
    they're working on. Every probe is simultaneously an assessment and a
    learning event.

3.  **Tutoring:** Optionally teaches users through adaptive Socratic
    dialogue, guided by the knowledge graph's estimate of their Zone of
    Proximal Development. Assessment and teaching are the same activity
    viewed from two perspectives.

4.  **Reporting:** Maintains a Bayesian knowledge graph with posterior
    distributions over mastery per concept, visible to the individual, their
    manager/professor, and the organization.

3.2 What Entendi Is Not

-   Not a replacement for existing AI tools (Copilot, Claude, Cursor, etc.)

-   Not a plagiarism detector or AI-content classifier

-   Not a restrictive gatekeeper that blocks tool usage

-   Not a surveillance tool --- it measures comprehension, not productivity,
    time-on-task, or response speed

-   Not a proctoring system --- it uses no timing analysis, keystroke
    logging, or behavioral biometrics

4.3 Entendi Interrogator

The Interrogator generates contextual Socratic probes. It operates as the
**inner loop** of an Intelligent Tutoring System architecture.

4.3.1 Dual-Loop Architecture

Entendi separates two concerns following decades of ITS research (VanLehn
2006, Carnegie Learning MATHia):

**Outer Loop (Algorithmic --- Knowledge Engine):**
- Selects WHAT to probe/teach next
- Driven by the knowledge graph and Bayesian model
- Identifies the Zone of Proximal Development: concepts whose prerequisites
  are mastered but which are not yet mastered themselves
- Uses Fisher information from IRT to pick the most informative concept
- Respects curriculum coherence and prerequisite ordering

**Inner Loop (LLM-Powered --- Interrogator/Tutor):**
- Decides HOW to probe/teach the selected concept
- Conversational scaffolding, Socratic questioning, hint escalation
- Evaluates responses on a 0-3 rubric
- Adapts dialogue strategy based on the student's responses

4.3.3 Depth Probing

Probes are not one-shot. Each probe initiates a **depth chain** where
follow-up questions build on the user's specific answers:

  -----------------------------------------------------------------------
  **Depth**   **Level**           **What It Tests**
  ----------- ------------------- -----------------------------------------
  0           Recall              Can name/describe the concept

  1           Comprehension       Can explain purpose and why it applies
              + Context           HERE (references user's codebase)

  2           Application         Can predict behavior under novel
              + Analysis          conditions ("what if we changed X?")

  3           Evaluation          Can compare alternatives, predict failure
              + Synthesis         modes, argue trade-offs given specific
                                  architecture
  -----------------------------------------------------------------------

Confidence is not additive. Each depth level's response is scored on the
0-3 rubric (Section 4.3.4) and fed into the Graded Response Model (Section
5.1) as a Bayesian observation. Deeper probes naturally correspond to
higher-discrimination items in the GRM --- a correct answer at depth 3
produces a larger posterior shift than a correct answer at depth 0, because
the GRM's discrimination parameter a is higher for more challenging probes.

Each depth level references increasingly specific details of the user's
actual codebase and decisions, making the probe progressively harder to
answer without genuine understanding of the work.

4.3.4 Response Evaluation

User responses are evaluated on a 4-level rubric by the LLM backend:

  -----------------------------------------------------------------------
  **Level**   **Label**           **Criteria**
  ----------- ------------------- -------------------------------------------
  0           No Understanding    Cannot explain at all, or gives an answer
                                  that is clearly parroted from documentation
                                  without comprehension. Vague, generic, or
                                  irrelevant.

  1           Surface             Can describe what something does but not
                                  why it's used or how it works. No evidence
                                  of reasoning about alternatives or failure
                                  modes.

  2           Functional          Can explain purpose, basic mechanics, and
                                  identify at least one tradeoff or failure
                                  mode. Demonstrates causal reasoning.

  3           Deep                Can reason about alternatives, predict
                                  failure modes, explain when NOT to use
                                  this approach, and connect to broader
                                  architectural context. Demonstrates
                                  transfer.
  -----------------------------------------------------------------------

Each rubric score maps to an observation in the Graded Response Model
(Section 5) and triggers a Bayesian update to the user's mastery posterior
for the relevant concept(s).

4.3.5 Escalating Scaffold

When the Interrogator detects struggle, it escalates support rather than
repeating the same question:

    Level 0: Observe --- let the user think (productive struggle)
    Level 1: Socratic question targeting the specific misconception
    Level 2: Targeted hint (narrower than a full explanation)
    Level 3: Partial explanation with a gap for the user to fill
    Level 4: Full explanation (the escape hatch)

    Rules:
    - If the user explicitly asks "just explain it" -> skip to Level 4
    - If the user has struggled through 3+ attempts -> auto-escalate
    - After Level 4 -> verify with a transfer question on a related concept
    - A Level 4 explanation still counts as a learning event and updates
      the knowledge graph (with a lower score)

This follows the Evidence-Decision-Feedback framework (Stanford SCALE, 2026)
and addresses the finding that purely Socratic approaches frustrate users
without escape valves.

4.3.6 Anti-Gaming Design

Entendi's integrity model is based on **depth, context, and consistency**
--- never on surveillance or timing.

**Primary: Depth Probing**
Follow-up chains go 3-4 levels deep, with each level referencing the user's
specific answer and codebase. Parroted answers from external LLMs break down
at depth 2-3 because:
- The user must copy-paste increasing context (original question, their
  previous answers, codebase details)
- Each follow-up is unique to their specific response
- Maintaining coherence across the chain requires genuine understanding

**Secondary: Consistency Checks**
The knowledge graph tracks understanding over time. Inconsistencies trigger
re-probing (not accusation):
- High probe scores but basic LLM queries on the same concept
- Claimed deep knowledge but avoidance of the concept in code
- Sudden competence jumps without corresponding learning activity

**Tertiary: Context Binding**
Probes reference specific file paths, function names, recent commits, and
architectural decisions from the user's codebase. This requires intimate
familiarity with the work that is hard to quickly feed to an external LLM.

**Quaternary: Transfer Tasks**
Probes ask for novel application (Bloom's higher levels: analyze, evaluate,
create) rather than recall. Research shows LLMs perform significantly better
at lower taxonomy levels and struggle at the top --- so transfer tasks are
inherently harder to game.

**High-Stakes: Collaborative Verification**
For certifications, production deployments, or thesis submissions, Entendi
flags for an AI-mediated oral defense (following the Georgia Tech Socratic
Mind model) or human peer review.

**The Convergence Hypothesis (to be empirically validated):**
If someone successfully "games" a depth-3 context-specific probe chain by
consulting an external LLM, the effort required --- copying context,
understanding the LLM's answer well enough to paraphrase it, handling
follow-ups that reference their specific previous answers --- may constitute
genuine engagement with the material. The design goal is to calibrate probe
depth so that **gaming and learning converge**: the minimum effort to game
exceeds the effort to simply learn. This is supported by Bjork's desirable
difficulties research (retrieval under challenge strengthens encoding) but
has not been empirically validated in this specific context. Validation
plan: have users deliberately attempt to game depth-3 probes using external
LLMs, then independently assess whether their understanding improved.

**What Entendi explicitly does NOT use:**
- Latency analysis (timing how long answers take)
- Behavioral biometrics (typing patterns, mouse movement)
- Screen recording or keystroke logging
- Comparison of response patterns between users

These are excluded not just for ethical reasons but because they are
statistically unreliable. Meta-analysis of 319 studies shows ADHD
individuals exhibit 0.71 SD more reaction time variability. Autistic
individuals score ~1 SD below mean on processing speed while showing
strengths in reasoning. Timing-based detection systematically punishes
neurodivergent users and violates ADA guidelines.

4.4 Entendi Tutor

The Tutor is a new component that transforms Entendi from pure assessment
into an adaptive learning system. Assessment and teaching are the same
activity: every probe teaches through the testing effect, and every
explanation assesses through the student's questions.

4.4.1 When the Tutor Activates

The Tutor activates in two modes:

**Reactive mode:** When a user fails a probe (rubric score 0-1), the Tutor
offers to teach the concept rather than just recording the gap. The user can
accept or defer.

**Proactive mode:** When the user explicitly requests teaching ("teach me
about MCMC") or when the system detects the user is working with a concept
they haven't mastered but need for their current task.

4.4.2 The Four-Phase Dialogue

Following SocraticLLM (NeurIPS 2024) and the learning-by-teaching paradigm:

    Phase 1 --- REVIEW: "What do you already know about X?"
    -> Assesses prior knowledge AND activates retrieval (dual-purpose)

    Phase 2 --- GUIDED DISCOVERY: "Given that, what do you think happens
    when Y?"
    -> Pushes toward ZPD through questions, not lectures

    Phase 3 --- RECTIFICATION: "Almost --- but consider Z. How does that
    change things?"
    -> Corrects misconceptions through targeted Socratic probing

    Phase 4 --- CONSOLIDATION: "Can you explain the full picture now?"
    -> Consolidates learning AND provides final assessment checkpoint

**Knowledge graph update policy for tutoring sessions:**

    Phase 1 response: Scored on the 0-3 rubric. Generates a Bayesian
    observation that updates the prior (this is the "before teaching"
    measurement).

    Phases 2-3: NOT scored. These are the teaching interaction. No
    knowledge graph updates during active instruction.

    Phase 4 response: Scored on the 0-3 rubric. Generates a second
    observation (the "after teaching" measurement). This observation
    carries a "tutored" flag that modifies its weight --- a score of 3
    after tutoring is evidence of learning, but weaker evidence of
    independent mastery than a score of 3 on an unprompted probe.

This design preserves the diagnostic value of Phase 1 (we learn what the
user knew BEFORE teaching) without penalizing them for pre-teaching gaps.

4.4.3 Resource Integration

When teaching, the Tutor can:
- Fetch and synthesize content from authoritative sources (textbooks,
  documentation, papers) via web search
- Generate worked examples using the user's actual codebase as context
- Create practice problems calibrated to the user's ZPD
- Reference the user's own code to ground abstract concepts in concrete
  experience

4.4.4 ZPD Estimation

The Zone of Proximal Development is estimated mathematically from the
knowledge graph:

    For each concept C:
      prerequisites_mastered = all(P(mastery) > threshold
                                   for p in C.prerequisites)
      concept_mastered = P(mastery) > threshold for C

      if prerequisites_mastered and not concept_mastered:
        C is in the ZPD (the "frontier")
      elif not prerequisites_mastered:
        C is beyond the ZPD (too hard --- teach prerequisites first)
      else:
        C is below the ZPD (already known)

    SELECT next_concept FROM frontier
    ORDER BY Fisher_information DESC, curriculum_coherence, recency

This ensures the Tutor always teaches what the user is **ready** to learn,
not what is next in a fixed sequence.

4.5 Entendi Knowledge Engine

The Knowledge Engine maintains a persistent, evolving Bayesian knowledge
graph for each user.

4.5.1 Graph Structure

The knowledge graph is a directed graph where:

-   **Nodes** represent concepts at the "topic" granularity: something you
    could spend 1-4 hours learning from scratch (e.g., "MCMC sampling",
    "React hooks", "dependency injection", "CAP theorem").

-   **Edges** represent typed relationships:

  -----------------------------------------------------------------------
  **Edge Type**           **Semantics**           **Constraint**
  ----------------------- ----------------------- -----------------------
  REQUIRES (prerequisite) Must understand A       DAG (no cycles).
                          before B                Enforced via asymmetry,
                                                  irreflexivity, and
                                                  transitive reduction.

  PART_OF                 A is a sub-concept of B DAG (strict hierarchy)

  RELATED_TO              A and B are associated  No constraint (symmetric)

  ALTERNATIVE_TO          A and B serve similar   No constraint (symmetric)
                          purposes

  USED_WITH               A is commonly used      No constraint
                          alongside B

  IS_EXAMPLE_OF           A is a concrete         DAG
                          instance of abstract B
  -----------------------------------------------------------------------

-   **Node state** is a posterior distribution over mastery (see Section 5),
    not an integer score. Each node also tracks: stability S, difficulty D,
    retrievability R(t), last assessment timestamp, and interaction history.

-   **Interaction logs** are attached to nodes: every time a concept is
    touched (via LLM interaction or probe response), the event is recorded
    in an append-only event store.

4.5.2 Concept Extraction Pipeline

Concepts are extracted from interactions through three parallel signal
sources:

**Signal A: Package/Dependency Analysis (highest confidence, lowest cost)**
Parse imports, package.json, requirements.txt, Cargo.toml, go.mod, etc.
Map packages to concepts via a maintained lookup table:

    "pymc"   -> [Bayesian Inference, MCMC, Probabilistic Programming]
    "react"  -> [React, Component Architecture, JSX, Virtual DOM]
    "prisma" -> [ORM, Database Schema, Query Building, Data Modeling]

~200 packages mapped covers ~80% of common interactions.

**Signal B: AST Analysis (code patterns)**
Use Tree-sitter for incremental AST parsing across 40+ languages. Detect:
- Language features used: async/await, generators, decorators, generics
- Design patterns: observer, factory, dependency injection
- Complexity signals: nesting depth, cyclomatic complexity
- API usage patterns: which standard library or framework APIs are called

**Signal C: LLM Structured Output (from prompts/responses)**
Two-pass extraction with a JSON schema:

    Pass 1 --- Entity Recognition:
    Extract all technical concepts mentioned or implied.

    Pass 2 --- Relation Extraction:
    Given entities, extract relationships (uses, requires, part-of,
    example-of).

    Output schema per interaction:
    {
      "concepts": [
        {
          "name": "canonical name",
          "specificity": "domain | topic | technique",
          "domain": "broader domain",
          "signals": ["what in the text indicated this"]
        }
      ],
      "primary_intent": "learning | building | debugging | designing",
      "apparent_familiarity": "novice | intermediate | advanced"
    }

4.5.3 Taxonomy Design: Seed + Emergent Hybrid

**Seed layer (top-down):** ~500 concepts bootstrapped from SWEBOK v4 Knowledge
Areas, ACM Computing Classification System, and common package ecosystem
mappings. This provides the stable backbone that makes knowledge graphs
comparable across users.

**Emergent layer (bottom-up):** New concepts discovered from interactions are
attached to the seed taxonomy via embedding-based similarity. When enough
concepts cluster in an area not covered by the seed, the system proposes a
new category for validation.

**Granularity heuristic:**
- **Domain** (broad): "Web Development", "Machine Learning" --- takes
  days/weeks to learn
- **Topic** (primary operating level): "React State Management", "Bayesian
  Inference" --- takes 1-4 hours to learn from scratch. Maps to a Stack
  Overflow tag or a tutorial section.
- **Technique** (specific): "useReducer with context", "MCMC sampling" ---
  takes minutes to learn once you know the topic

**Concept resolution:**
1. Embedding computation (all-MiniLM-L6-v2 or text-embedding-3-small)
2. Cosine similarity clustering (DBSCAN or hierarchical agglomerative)
3. LLM-as-judge cluster validation
4. Canonical name assignment
5. Taxonomy placement

**Prerequisite detection:**
1. LLM judgment to bootstrap: "Is understanding X prerequisite for Y?"
2. Interaction-derived signals to refine: if many users learn A before B,
   that's empirical evidence of prerequisite ordering (Entendi's unique
   data advantage)
3. DAG enforcement: reject cycles, enforce asymmetry and irreflexivity,
   apply transitive reduction

4.5.4 Concept Lifecycle

    1. DISCOVERED  --- First extracted from an interaction
    2. CANDIDATE   --- Awaiting clustering/normalization
    3. NORMALIZED  --- Matched to canonical form, placed in taxonomy
    4. VALIDATED   --- Confirmed by LLM-as-judge or expert review
    5. STABLE      --- High confidence, many observations, relationships
                       verified

Concepts can also be MERGED, SPLIT, or DEPRECATED as the taxonomy evolves.

5\. Probabilistic Model Specification

This section specifies the mathematical model that is the intellectual core
of Entendi. It combines Item Response Theory for observation, FSRS for
time decay, and Bayesian belief propagation on prerequisite graphs.

5.1 Observation Model: Graded Response IRT

Each concept c has item parameters estimated from population data:

    a_c ~ LogNormal(0, 0.5)       # discrimination: how well this concept
                                   # differentiates ability levels

    b_c,1 < b_c,2 < b_c,3        # threshold parameters for rubric
                                   # boundaries (0/1, 1/2, 2/3)

The probability of a user with mastery theta scoring at or above rubric
level k on concept c:

    P(score >= k | theta, a_c, b_c,k) = 1 / (1 + exp(-a_c * (theta - b_c,k)))

The probability of exactly score k:

    P(score = k | theta) = P(score >= k | theta) - P(score >= k+1 | theta)

    with boundary conditions:
    P(score >= 0 | theta) = 1
    P(score >= 4 | theta) = 0

This is Samejima's (1969) Graded Response Model. It naturally handles the
0-3 rubric without information loss (unlike dichotomizing for BKT). The
discrimination parameter a captures how well each concept's probes
differentiate between students of different ability levels.

5.2 Per-User Mastery State

Each (user, concept) pair maintains:

    theta_u,c      # mastery belief on the IRT scale (real line)
                   # represented as Normal(mu, sigma^2)
    S_u,c          # memory stability (FSRS)
    D_u,c          # concept difficulty for this user (FSRS)
    t_last_u,c     # timestamp of last assessment

Theta lives on the real line (-inf, +inf), matching the IRT convention where
both student ability and item difficulty share the same unbounded scale. The
Normal distribution is the standard choice for IRT ability parameters. The
mean mu is the point estimate of mastery. The variance sigma^2 captures
uncertainty --- high variance means we have little evidence about this
concept.

To interpret theta as a mastery probability (for display and gating):

    P(mastery) = logistic(theta) = 1 / (1 + exp(-theta))

    theta ~ 0   -> P(mastery) ~ 0.50 (maximum uncertainty)
    theta ~ 2   -> P(mastery) ~ 0.88 (likely mastered)
    theta ~ -2  -> P(mastery) ~ 0.12 (likely not mastered)

Initial prior for a new concept: Normal(0, 1.5^2), representing genuine
uncertainty. When prerequisite information is available, the prior mean
shifts (see Section 5.5).

5.3 Time Decay: FSRS Retrievability

Between assessments, knowledge decays following the FSRS power-law
forgetting curve:

    R(t, S) = (1 + factor * t / S)^(-decay)

    where:
    factor = 0.9^(-1/decay) - 1    # calibrated so R(S, S) = 0.9
    t = time since last assessment
    S = stability (time for R to drop to 90%)
    decay = learned parameter (default ~0.5)

When R(t) is low (long time since assessment), the effective prior regresses
toward the initial prior (uncertainty):

    theta_prior = Normal(0, 1.5^2)    # the initial "no evidence" prior

The effective distribution blends the last posterior with the initial prior
by interpolating the distribution parameters:

    mu_eff    = R(t) * mu_posterior    + (1 - R(t)) * mu_prior
    sigma_eff = R(t) * sigma_posterior + (1 - R(t)) * sigma_prior

When R(t) ~ 1 (recent assessment), the effective state is close to the
posterior. When R(t) ~ 0 (very stale), the state regresses toward genuine
uncertainty. This means stale knowledge is treated with appropriate
skepticism --- a mastery estimate from 6 months ago carries less weight
than one from yesterday.

5.4 Bayesian Update

When a user receives rubric score s on concept c at time t:

    # 1. Compute time-decayed effective mastery
    R = R(t - t_last, S_u,c)
    mu_eff, sigma_eff = decay_prior(mu_u,c, sigma_u,c, R)

    # 2. Compute GRM likelihood
    L(s | theta) = P(score = s | theta, a_c, b_c)
    # This is the probability of observing rubric score s given ability theta

    # 3. Bayesian update (approximate)
    # The GRM likelihood is not conjugate to the Normal prior, so we use
    # Laplace approximation: find the posterior mode and curvature.
    #
    # theta_MAP = argmax [ log L(s | theta) + log Normal(theta | mu_eff, sigma_eff^2) ]
    # sigma_posterior^2 = -1 / (d^2/dtheta^2 [ log posterior ] at theta_MAP)
    #
    # For Phase 0, a simpler update rule suffices: shift mu by a step
    # proportional to the surprise (observed score vs. expected score),
    # scaled by sigma (uncertain beliefs update more). See Section 13.1.

    # 4. Update FSRS stability and difficulty
    grade = map_rubric_to_fsrs(s)  # 0=Again, 1=Hard, 2=Good, 3=Easy
    S_new = fsrs_stability_update(S, D, grade, R)
    D_new = fsrs_difficulty_update(D, grade)

    # 5. Record new state
    mu_u,c = mu_posterior
    sigma_u,c = sigma_posterior
    S_u,c = S_new
    D_u,c = D_new
    t_last_u,c = t

5.5 Prerequisite Propagation

When mastery of concept c is updated, beliefs propagate through the
prerequisite graph:

**Forward propagation (mastery of prerequisite -> update successors):**

When a user demonstrates mastery of concept c, we propagate a prior boost
to concepts that DEPEND ON c (i.e., concepts for which c is a prerequisite):

    For each concept s where s REQUIRES c:  # s depends on c
      mu_s += boost * max(0, mu_c_posterior - mu_c_prior)
      # boost attenuates with graph distance (e.g., boost = 0.3 / depth)
      # Only positive updates propagate forward (mastery, not failure)

Example: mastering "probability distributions" (c) shifts the prior upward
for "Bayesian inference" (s), because s REQUIRES c.

**Inconsistency detection (not hard constraint):**

If a child concept has higher estimated mastery than its prerequisite, this
is treated as a **signal to re-probe**, not as a reason to silently adjust
scores. This is important because it IS empirically possible to understand a
downstream concept while being shaky on its prerequisite (e.g., deep React
hooks knowledge with weak JavaScript closures understanding).

    For each (parent, child) in REQUIRES edges:
      if E[theta_child] > E[theta_parent] + margin:
        flag parent for re-probing at next opportunity
        # The inconsistency may reveal:
        # (a) the user genuinely has a gap in the prerequisite, or
        # (b) the prerequisite assessment was unreliable (bad probe, bad day)
        # Re-probing resolves which case applies

This avoids the pedagogically unsound behavior of force-downgrading
demonstrated mastery, which would violate the "Right to explanation"
principle (Section 8.1) and feel punitive to users.

Propagation follows SKT's (Tong et al., 2020) design: unidirectional on
the prerequisite DAG (prerequisites -> successors, never backward). This
prevents oscillation and aligns with pedagogical intuition.

5.6 Adaptive Probe Selection

The next concept to probe is selected by maximizing expected information
gain. For the GRM, Fisher information at ability theta for concept c is:

    I_GRM(theta, c) = a_c^2 * sum_{k=1}^{3} [
      (P*_k - P*_{k+1})'^2 / (P*_k - P*_{k+1})
    ]

    where P*_k = P(score >= k | theta) are the cumulative probabilities.

For computational convenience in Phase 0, we use the 2PL approximation:

    I_approx(theta, c) = a_c^2 * P(theta) * (1 - P(theta))

    where P(theta) = logistic(a_c * (theta - b_c,2))
    (using the middle threshold as the representative difficulty)

This approximation preserves the ranking of concepts by informativeness.
The full GRM formula should be implemented by Phase 1.

Probe selection:

    next_concept = argmax_c [ I(theta_u,c) * novelty_weight(c)
                              * (1 - R(t, S_u,c)) ]

This selects concepts where:
- The system is most uncertain about mastery (high Fisher information)
- The concept is novel or recently encountered (high novelty weight)
- Knowledge may have decayed (low retrievability)

The result is that probes are never wasted on concepts the system already
confidently knows you understand, and concepts you're likely forgetting
get re-assessed at the right time.

6\. User Experience Flows

6.1 Developer Daily Flow

1.  **Work normally:** Developer uses Cursor/Copilot/Claude Code as usual.
    Entendi Observer runs through the appropriate adapter.

2.  **Contextual probe appears:** After installing a new package, a probe
    appears: "You just added zod. Quick question --- why schema validation
    at the boundary rather than deeper in the stack?"

3.  **Developer responds:** Types a brief answer. Takes as long as they
    need --- timing is never measured.

4.  **Depth chain (if needed):** Good answer at depth 0 may trigger a
    depth 1 follow-up referencing their specific code. The chain continues
    until confidence is sufficient or the user requests the explanation.

5.  **Bayesian update:** Entendi updates the posterior for "schema
    validation", "zod", and related concepts. FSRS updates stability.

6.  **Continue working:** Next probe is selected by Fisher information.
    May not come for 30+ minutes if the developer is in familiar territory.

7.5 Future Protocol Dependency: MCP Interceptors (SEP-1763)

The MCP specification has a proposal (SEP-1763) for Interceptors that would
enable transparent middleware patterns --- intercepting, validating, and
transforming messages at extension points including tool discovery, tool
invocation, and LLM interactions.

If SEP-1763 is adopted, Entendi's per-tool adapter architecture could be
replaced with a single standardized interceptor that works across all
MCP-enabled clients. The current adapter architecture is designed to be
replaceable: all adapters emit the same unified event schema, so the
downstream engine is adapter-agnostic.

**Do not design around SEP-1763 until it ships.** The per-tool adapter
approach works today.

8\. Privacy, Ethics & Governance

Entendi captures sensitive data about how people think and work. This
section is not an afterthought --- it's a foundational design constraint.

8.1 Data Ownership

-   **Individual data belongs to the individual.** Users can export or
    delete their knowledge graph at any time.

-   **Organizations see aggregated views by default.** Drill-down to
    individual probes requires explicit policy agreement.

-   **Interaction logs (raw prompts/responses) are never shared with
    managers.** Only derived comprehension posteriors and knowledge graph
    states are reportable.

-   **Right to explanation:** Users can see exactly why a concept was scored
    at a particular level --- the prior, the probe response, the rubric
    evaluation, and the posterior update. The Bayesian model is fully
    transparent.

-   **Right to challenge:** Users can request re-assessment of any concept.
    A new probe chain is generated and the posterior is updated based on the
    new evidence.

8.2 Anti-Surveillance Safeguards

-   Entendi measures comprehension, not effort, time, or productivity

-   **No timing signals whatsoever.** Response speed has zero weight in
    confidence calculations, scoring, or flagging. This is a hard
    architectural constraint, not a policy choice.

-   No keystroke logging, screen recording, mouse tracking, or behavioral
    biometrics

-   Probe responses are evaluated purely for conceptual understanding, not
    linguistic fluency, response length, or formatting

-   Managers cannot see what prompts a user sent to their AI tools --- only
    what concepts were involved and the comprehension posterior

-   Organizations must define a written policy for how Entendi data is used
    before deployment

8.3 Ethical Considerations

-   **Neurodivergence:** All assessment is untimed. Probe delivery and
    response formats are configurable (written, voice, code snippets,
    diagrams). Response patterns are never compared between users. The
    probabilistic model estimates mastery from content, never from
    behavioral signals.

-   **Language:** Non-native speakers are not penalized for expression
    quality. The LLM evaluator is prompted to assess conceptual
    understanding independently of linguistic fluency.

-   **Anxiety:** Framing matters. Entendi never uses language like "test,"
    "verify," or "check." Instead: "Let's explore this further," "Here's a
    related challenge," "What would you do if..." The UX emphasizes growth
    tracking, not evaluation.

-   **Power dynamics:** Clear policies must prevent Entendi data from being
    used in performance reviews without the employee's explicit consent and
    full context. The knowledge graph shows learning trajectory, not
    performance. A developer who starts with gaps but grows rapidly is
    demonstrating exactly the behavior the system rewards.

-   **Gaming reframe:** If a user engages deeply enough with a concept to
    pass a depth-3 context-bound probe --- even by consulting external
    resources --- they have learned. The system treats this as success,
    not gaming. The design goal is that the minimum effort to "game"
    Entendi exceeds the effort to simply learn the material.

11\. Open Questions

**Answered from v0.1:**

1.  ~~Evaluation model: same or separate LLM?~~ **Separate is
    recommended.** Using the same LLM creates a conflict of interest (the
    tool being evaluated generates the evaluation). A separate, smaller
    model (e.g., fine-tuned Llama or Haiku) for probe generation/evaluation
    reduces cost and eliminates the conflict.

2.  ~~Gaming surface area?~~ **Addressed via depth probing + convergence
    principle.** The design ensures that the effort to game exceeds the
    effort to learn. Latency analysis removed entirely.

3.  ~~Organizational taxonomy design?~~ **Hybrid: seed + emergent.**
    SWEBOK + ACM CCS backbone, with bottom-up emergence from interactions.

**Remaining open questions:**

4.  **Probe intrusiveness calibration:** The novelty-based probe frequency
    (~5% routine to ~80% critical) needs empirical validation. A/B testing
    across user populations is needed to find the optimal frequency curve.

5.  **Regulatory landscape:** In some jurisdictions (especially EU with
    works councils), even comprehension monitoring may require explicit
    consent frameworks. GDPR implications for knowledge graph data need
    legal review per-market.

6.  **IRT parameter cold start:** The Graded Response Model needs item
    parameters (discrimination, thresholds) per concept. For new concepts
    with no population data, what priors should we use? Options: (a) global
    defaults from similar concepts, (b) LLM-estimated difficulty, (c)
    uninformative priors with rapid updating.

7.  **Cross-tool identity unification:** When a user uses Claude Code,
    Cursor, and ChatGPT, their interactions are captured by different
    adapters. How do we unify these into a single knowledge graph? Requires
    an auth/identity layer.

8.  **Tutor quality evaluation:** How do we measure whether the Tutor is
    actually teaching effectively? MathTutorBench (EMNLP 2025) provides a
    benchmark framework, but we need Entendi-specific metrics: does a
    tutoring session result in higher probe scores on subsequent
    assessments of the same concept?

9.  **Population-level concept difficulty estimation:** With enough users,
    Entendi can empirically estimate which concepts are hardest (highest
    failure rates, lowest average mastery). Should this feed back into
    curriculum design recommendations for organizations?

10. **Model fairness audit:** The probabilistic model must be audited for
    demographic bias. Do certain populations systematically receive lower
    mastery estimates? If the LLM evaluator has language or cultural biases,
    these propagate into the knowledge graph.

13.1 Phase 0 Simplified Update Rule

The full GRM + Laplace approximation is deferred to Phase 1a. For Phase 0,
use a simpler update inspired by Elo ratings:

    # After observing rubric score s (0-3) on concept c:
    expected_score = 3 * logistic(mu_u,c)    # expected score given current ability
    surprise = s - expected_score             # positive = better than expected
    K = sigma_u,c^2 / (sigma_u,c^2 + noise)  # Kalman gain: uncertain beliefs update more

    mu_u,c += K * surprise
    sigma_u,c *= sqrt(1 - K)                  # uncertainty decreases with each observation

This converges to reasonable mastery estimates and can be replaced with the
full GRM model without changing the rest of the system (the downstream
consumers only use mu and sigma).

13.2 LLM Evaluation Reliability

The entire system's integrity depends on the LLM backend reliably scoring
responses on the 0-3 rubric. This is a known challenge.

**Known risks:**
- LLM evaluators show leniency bias (tendency to score higher than human raters)
- Evaluation varies across LLM providers and model versions
- Rubric scores can be sensitive to the student's phrasing rather than
  conceptual content
- Linguistic fluency and verbosity can inflate scores for non-conceptual reasons

**Calibration protocol:**

1. **Calibration set:** Maintain a human-labeled set of ~200 (probe, response,
   score) triples across the rubric spectrum. Re-evaluate these on every LLM
   backend change or model upgrade. Target: Cohen's kappa >= 0.7 with human
   labels.

2. **Confidence score:** The evaluator outputs a confidence alongside each
   rubric score. Responses with confidence below a threshold are flagged for
   re-probing rather than treated as reliable observations.

3. **Evaluation prompt engineering:** The evaluation prompt explicitly
   instructs: "Evaluate conceptual understanding only. Ignore grammar,
   fluency, verbosity, and formatting. A terse but correct answer scores
   higher than a verbose but vague one."

4. **Periodic drift detection:** Track the distribution of rubric scores over
   time. If mean scores drift upward (leniency creep) or downward, flag for
   recalibration.

5. **Dual evaluation (Phase 2+):** For high-stakes assessments, two
   independent LLM evaluations with disagreement sent to human review.

15\. Success Metrics

**Product-level KPIs:**

- **Probe completion rate:** % of probes answered (vs. dismissed/ignored).
  Target: >70% for novel concepts, >50% overall.
- **Knowledge graph coverage:** % of concepts encountered that have at least
  one probe assessment. Target: >60% within 30 days of use.
- **Mastery growth rate:** Average rate of posterior improvement per concept
  per week. Indicates the system is driving learning, not just measuring.
- **Independent validation correlation:** Correlation between Entendi mastery
  scores and independent assessment (e.g., coding interviews, exams, code
  review quality). This is the ultimate metric. Target: r > 0.5.
- **User retention (individual tier):** 30-day retention rate. Target: >40%.
- **NPS for probe experience:** Do users find probes useful or annoying?
  Target: NPS > 20.

**Model-level metrics:**

- **Rubric evaluation accuracy:** Cohen's kappa between LLM evaluator and
  human labels on calibration set. Target: >= 0.7.
- **Predictive validity:** Can the model predict probe scores on future
  assessments? Track AUC for predicting "will the user score 2+ on the next
  probe for this concept?"
- **Concept extraction precision:** % of extracted concepts that are
  correctly identified and placed in the taxonomy. Target: >80%.
- **Probe informativeness:** Average information gain per probe (reduction in
  posterior entropy). Higher = better probe selection.

**Organizational metrics (for customers):**

- **Comprehension debt ratio:** Ratio of concepts used in code to concepts
  with demonstrated mastery. Formally: |{concepts in codebase}| /
  |{concepts with P(mastery) > 0.7}|. A ratio > 2.0 indicates significant
  comprehension debt.
- **Knowledge concentration risk:** Number of concepts with only one team
  member at P(mastery) > 0.7. Higher = more bus factor risk.
- **Onboarding velocity:** Time for new hires to reach P(mastery) > 0.5 on
  core team concepts.

16\. Unit Economics and LLM Cost Management

Each probe requires LLM calls for generation and evaluation. At scale, this
is the primary cost driver.

**Cost per probe (estimated, Phase 0-1):**

    Probe generation:    ~500 input tokens + ~200 output tokens
    Response evaluation: ~800 input tokens + ~150 output tokens
    Concept extraction:  ~400 input tokens + ~100 output tokens (per interaction)

    Using Claude Haiku or GPT-4o-mini at ~$0.25/1M input, ~$1.25/1M output:
    Cost per probe cycle: ~$0.001 (generation + evaluation)
    Cost per interaction observation: ~$0.0003 (concept extraction)

**Cost per user per month (estimated):**

    Assume: 15 probes/day * 20 working days = 300 probes/month
    Assume: 50 interactions/day * 20 days = 1000 observations/month

    Probe cost:       300 * $0.001 = $0.30/month
    Extraction cost:  1000 * $0.0003 = $0.30/month
    Total LLM cost:   ~$0.60/user/month

    At $15/user/month (Team tier): ~96% gross margin on LLM costs
    At $5/student/semester (~$1.25/month): ~52% gross margin

These estimates assume cheap models (Haiku/4o-mini). Using larger models for
evaluation would increase costs 5-10x. The architecture should allow model
selection per function: cheap models for concept extraction, more capable
models for probe generation and evaluation.

**Cost reduction strategies:**
- Cache common probe patterns and adapt rather than generate from scratch
- Batch concept extraction (process last N interactions together)
- Use embedding similarity to skip extraction for near-duplicate interactions
- Reduce probe frequency for routine concepts (already built into the model)
